Gameplay & UI

What I worked on

  • Built the main number-matching loop: selection, pairing, scoring, hints, win/lose
  • Implemented a scrollable board that stays smooth even with many rows on screen
  • Connected in-game flow to menus, popups, missions, shop, and leaderboard
  • Integrated ads, in-app purchases, and remote tuning without breaking game pacing
  • Managed downloadable content and runtime loading to control memory on budget devices

Challenge & impact

Challenge: Clearing many rows at once caused stutter on low-end phones because too many UI elements were active at the same time.

What I did: Used a virtualized scroll list so only visible tiles are rendered, combined with staged animations and careful asset cleanup.

Result: Smoother scrolling and more stable performance during heavy clear sequences.

What is OSA? OSA (Optimized ScrollView Adapter) is a Unity UI technique similar to a virtual list: it only creates and updates the tiles currently visible on screen instead of building the entire board at once. That makes large, scrollable puzzle boards practical on mobile without wasting memory or CPU.

  3. UI layer — virtualized grid, popups, feedback effects

  5. Monetization — ad timing, rewarded offers, IAP shop flow

How I Work

Production Approach

Patterns I apply across shipped titles — explained in terms of player experience and production needs, not internal codebase details.

Remote content

  1. Download new levels or photo packs when needed
  2. Load only what the current screen requires
  3. Release assets after use to free memory

Keeps app size smaller and lets live content update without forcing a store release every time.

Decoupled game flow

  1. Gameplay event — e.g. level complete
  2. UI reacts — popup, score, buttons
  3. Other systems — sound, analytics, ads

Gameplay, UI, audio, and monetization stay independent so features are easier to change safely.

Ads with pacing

  1. Trigger point — level end, pause, reward offer
  2. Rules check — cooldown & frequency caps
  3. Show ad only when it fits the session flow

Monetization is tied to natural break points so it supports retention, not just impressions.

Clear game states

  1. Playing — input active, timer running
  2. Paused / Popup — input blocked, UI on top
  3. Win or Lose — results, ads, retry or continue

Predictable state handling prevents bugs like input leaking through popups or double-triggered rewards.
